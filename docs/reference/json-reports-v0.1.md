# JSON Reports v0.1

This page documents the JSON reports currently produced by OrbitFabric v0.1.0 development preview.

OrbitFabric JSON reports are generated artifacts intended for:

- CI inspection;
- automated validation workflows;
- reproducible scenario evidence;
- future integration tooling.

They are not the source of truth.

The source of truth remains the Mission Model YAML and scenario YAML.

---

## Stability status

The JSON report structures documented here are development-preview contracts.

They are intended to be stable enough for v0.1/v0.2 workflows, but they are not a v1.0 compatibility promise.

Future versions may introduce:

- explicit report schema versions;
- JSON Schema definitions;
- additional metadata fields;
- richer diagnostic structures;
- compatibility checks between tool version and Mission Model version.

---

## Version fields

OrbitFabric JSON reports include the OrbitFabric tool/package version in the top-level `version` field.

Example:

```json
{
  "tool": "orbitfabric-lint",
  "version": "0.1.0.dev0"
}
```

This identifies the OrbitFabric tool version that produced the report.

Mission Model version is represented separately.

For lint reports, the Mission Model version is currently exposed as:

```json
{
  "mission": "demo-3u",
  "model_version": "0.1.0"
}
```

The top-level `version` and `model_version` fields have different meanings.

See also:

```text
Reference -> Versioning Model
```

## Lint report

Generated by:

```bash
orbitfabric lint examples/demo-3u/mission/ \
  --json generated/reports/lint_report.json
```

Tool identifier:

```text
orbitfabric-lint
```

---

## Lint report top-level fields

| Field | Type | Meaning |
|---|---:|---|
| `tool` | string | Report producer. Current value: `orbitfabric-lint`. |
| `version` | string | OrbitFabric tool/package version that produced the report. |
| `mission` | string | Mission ID loaded from the Mission Model. |
| `model_version` | string | Mission Model version declared by the mission. |
| `result` | string | Machine-readable lint result. |
| `loaded` | object | Counts of loaded Mission Model domains. |
| `summary` | object | Count of findings by severity. |
| `findings` | array | List of lint findings. |

---

## Lint result values

Current lint report `result` values:

| Value | Meaning |
|---|---|
| `passed` | No error or warning findings. |
| `passed_with_warnings` | Warning findings exist, but no error findings exist. |
| `failed` | At least one error finding exists. |

---

## Lint `loaded` object

The `loaded` object reports how many Mission Model objects were loaded.

Current fields:

| Field | Meaning |
|---|---|
| `spacecraft` | Number of spacecraft records. Currently always `1` for a valid mission. |
| `subsystems` | Number of subsystem definitions. |
| `modes` | Number of operational modes. |
| `mode_transitions` | Number of mode transitions. |
| `telemetry` | Number of telemetry items. |
| `commands` | Number of command definitions. |
| `events` | Number of event definitions. |
| `faults` | Number of fault definitions. |
| `packets` | Number of packet definitions. |

---

## Lint `summary` object

The `summary` object reports finding counts by severity.

Current fields:

| Field | Meaning |
|---|---|
| `errors` | Number of `ERROR` findings. |
| `warnings` | Number of `WARNING` findings. |
| `info` | Number of `INFO` findings. |

---

## Lint finding object

Each entry in `findings` currently has this shape:

| Field | Type | Meaning |
|---|---:|---|
| `severity` | string | Finding severity. |
| `code` | string | Stable `OF-*` diagnostic or lint rule code. |
| `file` | string/null | File associated with the finding, when known. |
| `domain` | string/null | Mission Model domain associated with the finding, when known. |
| `object_id` | string/null | Object affected by the finding, when known. |
| `message` | string | Human-readable explanation. |
| `suggestion` | string/null | Suggested fix, when available. |

Example:

```json
{
  "severity": "WARNING",
  "code": "OF-CMD-005",
  "file": "commands.yaml",
  "domain": "commands",
  "object_id": "payload.start_acquisition",
  "message": "command should define timeout_ms",
  "suggestion": "Add timeout_ms to make command behavior testable."
}
```

## Simulation report

Generated by:

```bash
orbitfabric sim examples/demo-3u/scenarios/battery_low_during_payload.yaml \
  --json generated/reports/battery_low_during_payload_report.json
```

Tool identifier:

```text
orbitfabric-sim
```

---

## Simulation report top-level fields

| Field | Type | Meaning |
|---|---:|---|
| `tool` | string | Report producer. Current value: `orbitfabric-sim`. |
| `version` | string | OrbitFabric tool/package version that produced the report. |
| `mission` | string | Mission ID. |
| `scenario` | string | Scenario ID. |
| `result` | string | Machine-readable scenario result. |
| `summary` | object | Counts of executed scenario artifacts. |
| `timeline` | array | Rendered and structured timeline entries. |
| `events` | array | Events emitted during scenario execution. |
| `commands` | array | Commands dispatched during scenario execution. |
| `mode_transitions` | array | Mode transitions recorded during scenario execution. |
| `final_state` | object | Final mode and telemetry state. |
| `failed_expectations` | array | Failed scenario expectations. Empty when scenario passes. |

---

## Simulation result values

Current simulation report `result` values:

| Value | Meaning |
|---|---|
| `passed` | Scenario completed and all expectations passed. |
| `failed` | One or more scenario expectations failed. |

---

## Simulation `summary` object

Current fields:

| Field | Meaning |
|---|---|
| `events` | Number of events emitted during scenario execution. |
| `commands` | Number of commands dispatched during scenario execution. |
| `mode_transitions` | Number of mode transitions recorded during scenario execution. |
| `failed_expectations` | Number of failed expectations. |

---

## Timeline entry object

Each entry in `timeline` currently has this shape:

| Field | Type | Meaning |
|---|---:|---|
| `t` | number | Scenario time in seconds. |
| `time` | string | Rendered `MM:SS` time fragment. |
| `message` | string | Timeline message. |
| `rendered` | string | Fully rendered timeline line. |

Example:

```json
{
  "t": 5,
  "time": "00:05",
  "message": "COMMAND payload.start_acquisition -> ACCEPTED",
  "rendered": "[00:05] COMMAND payload.start_acquisition -> ACCEPTED"
}
```

---

## Event object

Each entry in `events` currently has this shape:

| Field | Type | Meaning |
|---|---:|---|
| `t` | number | Scenario time in seconds. |
| `event_id` | string | Event identifier. |
| `severity` | string | Event severity. |

---

## Command object

Each entry in `commands` currently has this shape:

| Field | Type | Meaning |
|---|---:|---|
| `t` | number | Scenario time in seconds. |
| `command_id` | string | Command identifier. |
| `status` | string | Command execution status. |
| `dispatch` | string | Dispatch mode. For example `MANUAL` or `AUTO`. |

---

## Mode transition object

Each entry in `mode_transitions` currently has this shape:

| Field | Type | Meaning |
|---|---:|---|
| `t` | number | Scenario time in seconds. |
| `from` | string | Source mode. |
| `to` | string | Target mode. |
| `reason` | string | Transition reason. |

---

## Final state object

The `final_state` object currently contains:

| Field | Type | Meaning |
|---|---:|---|
| `mode` | string | Final operational mode. |
| `telemetry` | object | Final telemetry values after scenario execution. |

Example:

```json
{
  "mode": "DEGRADED",
  "telemetry": {
    "obc.mode": "NOMINAL",
    "eps.battery.voltage": 6.7,
    "eps.battery.current": 0.4,
    "payload.acquisition.active": false,
    "radio.downlink.available": true
  }
}
```

## Current limitations

Current v0.1.0 development preview reports do not yet include:

- explicit report schema version;
- JSON Schema URL;
- report generation timestamp;
- source file paths;
- Git commit SHA;
- environment metadata;
- compatibility matrix;
- stable v1.0 compatibility guarantee.

These may be added later, but they are not part of the current report contract.

---

## Contributor rules

When changing report structure:

1. update this page;
2. update report tests;
3. keep version semantics aligned with the Versioning Model page;
4. avoid breaking existing fields without a deliberate design issue;
5. prefer additive changes during development preview.
